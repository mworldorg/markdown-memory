# Стандарт проектирования UI/UX Telegram-ботов (tg-bot-ui-ux)

Этот документ представляет собой единый стандарт проектирования интерфейсов и логики взаимодействия (UI/UX) для Telegram-ботов. Он призван обеспечить единообразие, профессиональный вид и защиту от типичных багов при автоматической генерации ботов ИИ-агентами.

---

## 1. Логика навигации и клавиатуры

Интерфейс строится по гибридному принципу: постоянная Reply-клавиатура для глобального меню и Inline-клавиатуры для контекстных действий.

### 1.1. Reply-клавиатура (Главное меню)
*   **Сценарий**: Используется **только** на самом верхнем уровне (Главное меню после `/start` или при нажатии кнопки сброса).
*   **Параметры**: Всегда `resize_keyboard=True` (для уменьшения высоты клавиатуры на мобильных) и `one_time_keyboard=False`.
*   **Сетка (Layout)**: Строго **по 2 кнопки в ряд** (сетка 2хN). Если кнопок нечетное количество, последняя растягивается на всю ширину.
*   **Именование**: Каждая кнопка содержит тематический эмодзи, пробел и существительное/действие с Большой буквы (например, `👥 Аккаунты`, `⚙️ Настройки`).

### 1.2. Inline-клавиатуры (Внутренние меню и списки)
*   **Именование кнопок действий**: Кнопка должна четко выражать действие по шаблону `{эмодзи} {Глагол в инфинитиве} {объект со строчной буквы}`. 
    *   *Правильно*: `➕ Добавить аккаунт`, `🗑 Удалить расписание`, `🚪 Выйти из чатов`.
    *   *Запрещено*: использовать абстрактные слова без контекста (`ОК`, `Далее`, `Да`, `Нет`, `Назад` без эмодзи).
*   **Сетка кнопок (Grid Layout)**:
    *   **1 кнопка в ряд**: Для основных действий в формах, длинных списков настроек.
    *   **2 кнопки в ряд**: Для компактных переключателей (например, `[Только joined] [Все ссылки]`) или выбора элементов.
    *   **3 кнопки в ряд**: Для плотных сеток (например, выбор месяцев, дней или компактная сетка статусов аккаунтов).
*   **Постраничная навигация (Pagination)**:
    *   Лимит элементов на страницу — **6-8 элементов**.
    *   Ряд управления пагинацией располагается в самом низу списка:
        `[ ⬅️ Назад ] [ {page}/{total} ] [ ➡️ Вперёд ]`
        Если навигация допускает глобальные действия над списком, они встраиваются по центру:
        `[ ⬅️ Назад ] [ 🔁 Обновить все ] [ ➡️ Вперёд ]`
*   **Кнопка возврата**: В самом низу любого inline-меню (кроме главного) должна быть кнопка возврата на предыдущий уровень: `⬅️ Назад` (callback_data: `back` или `parent_menu_id`).

### 1.3. Диалоги подтверждения (Confirmation UX)
*   Все деструктивные или важные действия (удаление, очистка, сброс) требуют двухэтапного подтверждения.
*   Бот присылает сообщение с описанием действия и последствий (`⚠️ Вы уверены, что хотите удалить {объект}? Это действие необратимо!`).
*   В клавиатуре выводятся ровно две кнопки в один ряд:
    `[ ✅ Да ] [ ❌ Отмена ]`
    Именно в таком порядке: подтверждение — слева, отмена — справа.
*   При нажатии `❌ Отмена` интерфейс возвращается в исходное состояние (сообщение редактируется обратно).

## 2. Оформление сообщений и верстка

Тексты сообщений должны легко сканироваться глазами. Форматирование должно быть безопасным для API Telegram.

### 2.1. Разметка (Parse Mode)
*   **Стандарт**: Использовать **HTML** (`parse_mode="HTML"`).
*   **Безопасность (Критично)**: Любые динамические данные, приходящие от пользователя или внешних API (имена, юзернеймы, логи, ошибки), должны быть принудительно экранированы через `html.escape()`.
*   **Запрет на MarkdownV2**: MarkdownV2 строго запрещен для вывода динамических данных, так как неэкранированные спецсимволы (`_`, `*`, `[`, `]`, `.` и др.) вызывают ошибку API `400 Bad Request` и ломают работу бота.

### 2.2. Правила верстки текста
*   **Заголовки**: Выделяются жирным текстом и начинаются с крупного эмодзи. Под заголовком располагается сплошная разделительная линия:
    ```html
    📊 <b>Статистика парсинга</b>
    ━━━━━━━━━━━━━━━━━━━━━━━━
    ```
*   **Одновременное копирование (Code Wrapping)**: Все системные ID, API-ключи, токены, пути к файлам и коды подтверждения оборачиваются в тег `<code>...</code>`. В клиентах Telegram это позволяет скопировать значение в буфер обмена в один тап.
*   **Логи и таблицы**: Выводятся внутри тега `<pre>...</pre>`, что сохраняет моноширинный шрифт и вертикальное выравнивание столбцов (через символ `|`) на любых экранах.
*   **Примечания и подсказки**: Оформляются курсивом `<i>...</i>`.
*   **Выравнивание колонок**: Использовать обычные пробелы для формирования табличной верстки. Использование знака табуляции `\t` запрещено, так как Telegram сжимает его.
*   **Лимиты списков**: Если список содержит более 50 элементов, выводить первые 40, а остаток указывать через троеточие `…` (один unicode-символ `U+2026`, а не три точки):
    ```
    • Элемент 1
    • Элемент 2
    … и еще 124 элемента
    ```

### 2.3. Статусные индикаторы (Status Badges)
Использовать эмодзи в качестве однозначных статусных иконок:
*   `🟢` — активно / в работе / успешно.
*   `⏸️` — на паузе / остановлено пользователем.
*   `⏳` — в ожидании / cooldown (например, FloodWait).
*   `🧊` / `🚫` — заморожен / спамблок.
*   `💀` — деактивирован / отключен.
*   `⚠️` — предупреждение / ошибка конфигурации.

## 3. Сценарии диалогов и UX-безопасность

Пользователь никогда не должен зависать в диалогах или ломать бота непредвиденным вводом.

### 3.1. Защита от зависания в FSM (FSM Menu Interceptor)
*   **Паттерн**: Если пользователь находится внутри состояния FSM (ожидание ввода текста/файла) и нажимает кнопку Reply-клавиатуры главного меню (например, `👥 Аккаунты`), бот должен перехватить это событие.
*   **Реализация**: Хендлер FSM сначала проверяет текст сообщения по списку кнопок главного меню. Если совпадение найдено:
    1. Сбрасывает состояние (`await state.clear()` или `reset_state`).
    2. Вызывает обработчик нажатой кнопки меню.
*   Это предотвращает баг, когда текст кнопки меню отправлялся в качестве ответа на вопрос формы.

### 3.2. Сброс и Отмена в формах
*   На каждом шаге FSM-формы на inline-клавиатуре должна присутствовать кнопка `❌ Отмена`.
*   Если на каком-то шаге требуется дать возможность сбросить настройку на дефолт (например, очистить описание), боту передается символ дефиса (`-` или `–`). Бот интерпретирует его как сигнал к сбросу значения на `None`.
*   Для массовой загрузки файлов поддерживать массив стоп-слов: `STOP_UPLOAD_WORDS = {"готово", "стоп", "/stop", "/cancel", "все", "хватит"}`.

### 3.3. Обработка ошибок ввода
*   При некорректном вводе (неверный формат, нечисловое значение) бот отправляет сообщение с описанием ошибки и примером корректного формата.
*   Бот **не сбрасывает** состояние FSM, позволяя пользователю сразу повторить ввод.
*   Если числовой параметр выходит за границы разумного диапазона, бот автоматически корректирует его до ближайшего лимита (например, `min(val, 100)`) и сообщает об этом пользователю.

### 3.4. Асинхронные операции и уведомления
*   **Всплывающие окна (Alerts)** `query.answer(text, show_alert=True)`: Использовать для критических ошибок, результатов проверок или подтверждений деструктивных действий.
    *   *Костыль копирования*: Если нужно показать ссылку или токен для копирования на мобильном, выводить его в `show_alert=True`. Из этого окна текст копируется долгим тапом.
*   **Всплывающие баннеры (Toasts)** `query.answer(text, show_alert=False)`: Использовать для подтверждения рутинных операций (`✅ Данные обновлены`, `🔄 Загрузка...`).
*   **Прогресс-бары**: Для операций длительностью более 10 секунд выводить визуальный прогресс-бар (`🟩🟩⬜️⬜️⬜️`). Если реальный прогресс неизвестен, использовать эмуляцию ("мягкий рост") с прибавлением 1-2% каждые 8-10 секунд, чтобы показать активность бота.
